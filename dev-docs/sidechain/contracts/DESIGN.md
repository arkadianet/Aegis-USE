## Theft-prevention invariants (what these contracts exist to enforce)

| Inv | Statement | Where enforced | v1 status |
|---|---|---|---|
| **I1 reserve** | `vault + unmerged receipts + unmerged fee ≥ notes + emission + pending burns` | cross-box accounting (off-chain metric + halt) + PegVault/Consolidator preserve-value | partial: on-chain contracts preserve per-tx value; the *global* inequality is an ops metric (I1 can't be a single-box invariant) |

| **I3 no double-redeem** | each Aegis burn id pays once | **DoubleRedeem** AvlTree insert-once | ✅ expressible |
| **no foreign-burn theft** | a payout corresponds to *its* burn; can't drain a box a burn doesn't match | PegVault binds payout to the UnlockIntent's `burn_id` + DoubleRedeem of *that* id; pays only from the singleton vault, never a per-user box | ✅ structure expressible; **C1 caveat below** |
| **V_cap** | `vault USE ≤ V_cap` | PegVault checks the post-spend vault token amount | ✅ expressible |
| **T_delay** | `HEIGHT ≥ UnlockIntent.start + T_delay` before payout | PegVault HEIGHT check vs UnlockIntent register | ✅ expressible |

## ⚠ The load-bearing open item — C1 (burn authenticity)

A PegVault payout must be authorized by a **real** Aegis burn. In v1 there is **no cryptographic proof on the Ergo side that the burn happened** — the burn is attested by the miner-posted `SideChainState` tip (pass 3: the payout must prove `burn_id → N` membership in the tip's **append-only** burn tree, so a fake burn now requires the tip key *and* leaves a permanent on-chain insert record — but the set is still TIP_PK-trusted). Under **U1-dogfood** this is *trusted*, bounded only by `V_cap` + `T_delay` (a lying tip → fake burn → drain is the C1 attack; the mitigation is the cap + delay + honest UI, NOT the contract). The contract therefore **cannot** be written to make peg-out trust-minimized in v1; it enforces the *ceremony* (UnlockIntent → delay → DoubleRedeem → capped payout from the singleton), not burn authenticity. Trust-minimization needs U1-strong (k-of-n attesters signing `(net, height, tip_digest, burn_id, N)` — S1) or SPV-in-consensus (S2). **This is stated in every contract's security block; do not deploy real value beyond `V_cap` on the v1 form.**

## Boxes / registers

### DepositReceipt (peg-in, many parallel boxes — no vault contention)
- **tokens[0]** = (USE id, `N`).
- **R4** `Coll[Byte]` = `sc_dest` (Aegis address payload) — immutable while locked.
- **R5** `Coll[Byte]` = memo (optional). **R6** `Coll[Byte]` = version/network id.
- **R7** `SigmaProp` = depositor key (refund path). **R8** `Int` = refund timeout height.
- **Spend paths:** (a) **consolidator merge** — consumed with the PegVault (identified by its NFT) present as OUTPUTS(0) absorbing this box's USE; **this is the mint commit-point** (peg.md §3.1) — the SC mints against *this* tx, not the lock; (b) **refund** — after `HEIGHT ≥ R8`, depositor sig, only if the box was never consolidated (which it can't have been — the box is consumed by (a)). Because (a) and (b) both spend the one box, **mint (needs (a)) and refund ((b)) are mutually exclusive by box-spend** — the refund-after-mint double-claim (I6) is closed structurally. The old `R8 ≥ creation + N_MINT + REFUND_MARGIN` race-floor is now redundant defense-in-depth rather than the interlock; it can be relaxed to a UX window. **SC-side obligation:** the mint used-set keys on the receipt `boxId`, but `verify_pegmint` must prove the **consolidation** tx (sc_dest/N read from the consumed receipt), not the lock.

### PegVault (singleton pooled reserve)
- **tokens[0]** = (vault NFT, 1) singleton; **tokens[1]** = (USE id, `vaultUSE`).
- **R4** `Coll[Byte]` = params hash / config (V_cap, USE id, T_delay, DoubleRedeem NFT, UnlockIntent script hash) — or these are inlined constants (v1 inlines).
- **Spend paths:** (a) **payout** — consumes a matching **UnlockIntent** (by burn_id, claimant, N) whose `start + T_delay ≤ HEIGHT`, updates **DoubleRedeem** with burn_id in the same tx, pays claimant `N − peg_out_fee`, keeps fee-worth in the vault, preserves the singleton NFT, keeps `vaultUSE_out ≤ V_cap`; (b) **consolidator top-up** — absorbs DepositReceipt/FeePot USE, singleton preserved, `≤ V_cap`.

### DoubleRedeem (burn-once ledger)
- **tokens[0]** = singleton NFT. **R4** `AvlTree` = set of consumed burn ids.
- **Spend:** insert the tx's `burn_id` (OUTPUTS' R5) into the tree via a proof; `insert` **fails if the key already exists** → double redeem blocked. Preserve NFT/value/script.

### UnlockIntent / FeePot / Consolidator — **pass 2** (see below).

### SideChainState (pass 3 — the C1 trust anchor, singleton tip-digest box)
- **tokens[0]** = (SIDECHAIN_STATE_NFT, 1) — minted once at deploy; token ids are first-input box ids, unforgeable.
- **R4** `Long` = Aegis sidechain height `h` — **strictly increasing**, jumps allowed (many Aegis blocks per Ergo block).
- **R5** `Coll[Byte]` = Aegis tip commitment at `h` (32-byte block id / state digest). **Unverified data in v1** (GAP-2).
- **R6** `AvlTree` = **burn tree**: key = `burn_id` (32 B), value = `longToByteArray(N)` (8 B). Genesis = empty tree, keyLength 32, valueLength Some(8), **enabledOperations = insert only**.
- **R7** `Int` = Ergo height of the last update (rate limit: ≤ 1 update per Ergo block).
- **Spend path (single): UPDATE** — `proveDlog(TIP_PK)` (v1 trusted miner-tip key — §C1) **and** structural (successor at OUTPUTS(0), same script, exactly the NFT, `value` non-decreasing) **and** `successor.R4 > SELF.R4` **and** `successor.R5.size == 32` **and** `successor.R7 == HEIGHT && HEIGHT > SELF.R7` **and** the **transition-constrained tree**:
  - ops posted (`getVar(0)` = `Coll[(Coll[Byte],Coll[Byte])]`, `getVar(1)` = proof): `successor.R6 == SELF.R6.insert(ops, proof).get`;
  - no ops: `successor.R6 == SELF.R6`.

  Every digest the box can ever hold is an **insert-descendant of the genesis empty tree**: even TIP_PK cannot swap in a foreign/fresh tree, delete/mutate a recorded burn (`insert` fails on an existing key ⇒ each burn's `N` is immutable once posted), or change tree params/flags (`==` on `AvlTree` compares digest **and** keyLength/valueLength/enabledOperations). `AvlTree` values are compared whole — no digest byte-slicing anywhere, which **sidesteps the pass-1 `0..32` vs `1..33` slice question** for this surface.
- **Read path:** UnlockIntent (at payout-spend time) requires `CONTEXT.dataInputs(0)` to carry the state NFT and proves `burnTree.get(burn_id, proof).get == longToByteArray(N)` — burn existence **and** amount, against the **live** box only (dataInputs must be unspent ⇒ no stale-tip replay).
- **Relation to DoubleRedeem (the AVL wiring):** two disjoint ledgers keyed by the same `burn_id` — SideChainState.R6 = *burned-set* (what Aegis burned; grows on tip updates), DoubleRedeem.R4 = *redeemed-set* (what the vault paid; grows on payouts). **DoubleRedeem.es is unchanged** — it never reads SideChainState, and its own transition was already insert-constrained in pass 1 (`OUTPUTS(1).R4 == SELF.R4.insert(...).get` — the same digest-swap defense pattern). A payout therefore needs `burn_id ∈ burned-set` (UnlockIntent dataInput lookup) **and** `burn_id ∉ redeemed-set` (DoubleRedeem insert succeeds): burned ⇒ payable, paid ⇒ never again.

## Peg-in lifecycle (mint commit-point = consolidation — see peg.md §3.1)
`user tx: [DepositReceipt (N USE + R4=sc_dest)] + [fee output → FeePot addr]` → `consolidator tx: [receipt(s)...] + [PegVault] → [PegVault' (+ΣN USE)]` (**consumes the receipt — the commit-point**) → (after `N_mint` Ergo confs *on the consolidation*) aegis-node PegMint note to `sc_dest` (read from the consumed receipt), mark `boxId` used. Refund only if a receipt is never consolidated (⇒ never minted). The consolidation is permissionless and the depositor is incentivised to run it (it triggers *their* mint).

## Peg-out lifecycle
`Aegis PegBurn(N)` → miner tip update inserts `(burn_id → N)` into SideChainState.R6 → `UnlockIntent tx (binds burn_id, N, claimant; starts T_delay)` → wait `T_delay` → `payout tx: [PegVault] + [UnlockIntent] + [DoubleRedeem] + dataInput [SideChainState] → [PegVault' (−N)] + [DoubleRedeem' (+burn_id)] + [claimant N−fee]` (UnlockIntent's spend proves `burn_id → N` membership in the live tip's burn tree; DoubleRedeem's insert-once blocks a second payout).

## Pass 2 — DONE (2026-07-12), all 5 contracts COMPILE (tree v3, repo Rust compiler)

**Critical siphon FIXED (in PegVault, where it belongs).** The vault now sums the USE of all consumed `DepositReceipt`/`FeePot` inputs (pinned by `RECEIPT_SCRIPT_HASH`/`FEE_POT_SCRIPT_HASH` via `INPUTS.fold`) and requires, per path:
- **TOP-UP:** `vaultOut == vaultIn + Σ(receipt USE) + Σ(feePot USE)` — diverting any consumed receipt/fee makes `vaultOut` too low → fail. Siphon closed in consolidation.
- **PAYOUT:** `receiptSum == 0 && feeSum == 0` — a payout can't smuggle receipts/fee to siphon them. Siphon closed in payout.

Conservation trace (why it's total, not partial): in top-up, token conservation forces every consumed receipt/fee's USE somewhere; the `==` requires the vault absorb exactly it, so nothing can be routed to an attacker output. In payout, no receipts are present, and `paidOut = vaultIn − vaultOut` is pinned to `N − fee` with the claimant getting exactly `paidOut` → inputs USE == outputs USE with no attacker room.

**Defense-in-depth added (review):** fee PINNED (`paidOut == N − max(floor, bps·N)`, closes the zero-fee leak); `vaultOut.tokens.size == 2` (no extra-token siphon); `T_delay` start = `intent.creationInfo._1` (unspoofable, replaces a prover register); DepositReceipt refund requires `R8 ≥ creation + N_MINT + margin` (refund-before-mint race defense; the authoritative interlock stays the SC used-set/I2).

**New contracts:** `UnlockIntent.es` (spendable only as INPUTS(1) with the vault at INPUTS(0) ⇒ every spend pays its own R6 claimant; anti-grief; double-post harmless via DoubleRedeem insert-once); `FeePot.es` (permissionless sweep into the vault; `feeSum`-accounted). Consolidator is the tx pattern enforced by the vault top-up + receipt/fee merge paths (no separate script). `DoubleRedeem.es` unchanged (reviewed sound).

## Pass 3 — DONE (2026-07-13), all 6 contracts COMPILE (tree v3, repo Rust compiler, same harness as passes 1–2)

**New: `SideChainState.es` (209 B)** — the singleton tip-digest box (see its section above). The two load-bearing design points:

1. **Transition-constrained burn tree (the digest-swap defense).** The successor's R6 is never merely "defined" (the upstream ErgoHack TODO left it exactly that — swappable): it must equal `SELF.R6.insert(ops, proof).get` (ops posted) or `SELF.R6` (no ops). `insert` verifies the proof against the OLD digest and *returns the new tree* — the only reachable digests are insert-descendants of the genesis empty tree. A swap to a fresh/foreign tree is not the insert-image of the old one (equality would require a blake2b digest collision); deletes/mutations are impossible (insert-only flags, pinned by `AvlTree ==`); re-inserting an existing `burn_id` fails, so a recorded burn's `N` is immutable.
2. **Verbatim-honest C1 trust statement:** v1 = **declared trusted-miner-tip key (`TIP_PK`) under `V_cap`** — NOT trust-minimized. The contract makes the trusted key *append-only and auditable* (every fake burn is a permanent on-chain insert record); it does not make burns provable. Do not deploy value beyond `V_cap`.

**Changed: `UnlockIntent.es` (81 → 159 B)** — payout-spend now additionally requires `CONTEXT.dataInputs(0)` = live SideChainState singleton (by NFT) and `R6.get(burn_id, getVar(0) proof).get == longToByteArray(N)`. Insert-only on the tip side makes this check *stable*: membership only grows, so a burn posted at intent time is still provable `T_delay` later. **`DoubleRedeem.es` unchanged** (preferred adaptation direction per plan: SideChainState/UnlockIntent adapted to it; its own R4 transition was already insert-constrained in pass 1). **`PegVault.es` untouched** — the pass-2 sum-accounting fix is load-bearing and reviewed; the burn check runs anyway in every payout because the vault pins `UNLOCK_INTENT_SCRIPT_HASH` at INPUTS(1) and an input's script executes when spent.

| # | Attack | Verdict | Killing check |
|---|---|---|---|
| A1 | **Forge or steal the singleton** — mint a lookalike NFT / re-script a fake state box readers will trust; or spend the real box and carry the NFT off to an attacker script | **Killed** | Token ids = first-input box ids (unforgeable; NFT minted once at deploy). Residual (accepted, C1): the updater can simply *stop updating* — the live box goes stale; that is a liveness failure, not a replay. |
| A3 | **AVL digest swap to a fresh empty tree** (erase history → double-redeem) — the crux | **Killed, both trees** | *Burned-set:* `successor.R6 == SELF.R6.insert(ops, proof).get` (or `== SELF.R6`) — a fresh/foreign digest is not the insert-image of the old tree; even TIP_PK can only append. | A4 | **Non-monotone height rollback** — repost a lower/equal `h` (with a pruned burn set) to "reorg away" burns | **Killed** | `successor.R4[Long].get > SELF.R4[Long].get` (strict) and the burn tree can't shrink regardless (A3). `successor.R7 == HEIGHT && HEIGHT > SELF.R7` blocks same-block thrash. Residual: a *genuine* Aegis reorg is unrepresentable — GAP-3 policy below. |
| A5 | **Update-path fee siphon** — fund update tx fees (or skim) from the state box's ERG endowment, bleeding it toward storage-rent death / spendability loss | **Killed** | `successor.value >= SELF.value` and `successor.tokens.size == 1` (no token stuffing/siphon). The updater funds fees from own inputs. |
| A6 | **Interaction with PegVault sum-accounting** — smuggle the state box into a vault tx to perturb `receiptSum`/`feeSum` or the payout structure | **Killed structurally** | As a dataInput it is not in `INPUTS` at all. PegVault then pays exactly `N − fee` against that `N`. |

A *wrong-box* proof fails the digest check; a proof for a different key fails the lookup. |
| A9 | **Fake-burn append via TIP_PK** (C1 itself) | **NOT killed — declared trust** | Out of contract reach in v1 by design: TIP_PK can insert a burn that never happened on Aegis and drain up to the vault balance, bounded by `V_cap` + `T_delay` (halt window) + the append-only audit trail (the fraudulent insert is permanent, attributable on-chain evidence). ## Pass-3 honest GAPs (cannot be expressed in ErgoScript v1 — recorded, not faked)

- **GAP-1 (=C1):** burn authenticity is trusted from `TIP_PK`. The contract enforces *append-only + auditable*, nothing stronger. v1 mitigation: `V_cap` + `T_delay` + fraud-halt + honest UI.
- **GAP-2:** the R5 tip commitment's *transition* is unverifiable on-chain — ErgoScript cannot verify Aegis headers/PoW or that R6's inserted burns are actually included in the R5-committed chain. R5↔R6 consistency is checked off-chain by aegis-node observers (fraud-halt on divergence). Needs U1-strong/SPV.
- **GAP-3 (Aegis reorgs):** strict height monotonicity + insert-only tree cannot represent a rollback. v1 policy: the tip only posts burns at ≥ `M_conf` Aegis depth; an inserted-then-rolled-back burn is a C1-class fraud event → halt. (Same posture as peg.md §7.)
- **GAP-4 (hot-tip contention):** a payout tx binds to the current state boxId as dataInput; a tip update between build and inclusion invalidates it (rebuild + retry; bounded by the 1-update-per-Ergo-block rate limit). Known should-fix: receipt tokens (peg.md).
- **GAP-5 (deploy ceremony is load-bearing):** genesis must mint exactly one state NFT and seed R6 = empty AvlTree, keyLength 32, valueLength Some(8), **insert-only flags**, R7 = deploy height, plus a rent endowment (`R_rent`). The contract preserves these; it cannot create them. **REVIEW-CONFIRMED (2026-07-13) this is the ONE-AND-ONLY chance:** the interpreter's insert success path returns the old tree with *only the digest replaced* (`method_call.rs:2766-2770`), so `keyLength`/`valueLengthOpt`/flags are **inherited immutably through every update** and are **unenforceable in-script at genesis** — a ceremony error (wrong keyLength, insert-disabled, wrong valueLength) is permanent and un-fixable. Verify the genesis box byte-for-byte before funding.
- **Note:** an update posting `getVar(0)` = empty ops coll either verifies to the unchanged tree or fails proof verification (verifier-dependent) — harmless in both cases and still TIP_PK-gated; the honest no-burns update uses the no-var branch (`newTree == oldTree`).

## Original finding (2026-07-13) — PegVault eager-ValDef bug (P1 correctness, NOT theft)

First real testnet deploy (lock + consolidation confirmed on Ergo testnet, txids in `test-vectors/testnet/peg/README.md`) surfaced a **runtime** bug that compilation and the semantic review both missed — *compilation ≠ correct evaluation*. Node rejected the clean consolidation (`vault + receipt` only) with:
`sigma.exceptions.InvalidType: Cannot getReg[Long](5): invalid type of value TestValue(Coll(97,101,…)) at id=5`.
**Cause:** `PegVault.es` `payoutOk` reads `intent.R5[Long].get` (twice), so the sigma compiler's CSE **hoists it into an eager top-level ValDef** that evaluates on *every* vault spend — including top-up/consolidation — and `.get` as `Long` **throws** when `INPUTS(1)` is a receipt carrying a `Coll[Byte]` memo at R5.
